# Semantic Enrichment Pipeline

The provenance fields are copied from the canonical DB, but the semantic fields (`summary`, `abstract`, `process`, `topics`, `people`, `clients`, `classification`, `data_sensitivity`, `sensitive_data_types`) are generated by an LLM at index time. This document describes that pipeline — the "process sotteso" to every conversation.

# Schema

## Stages

1. **Selection** — `kb_indexer.py` reads interactions from the canonical SQLite DB. Incremental mode processes only new/unindexed sessions; `--min-messages` skips trivial ones.

2. **Input filtering** — `prepare_messages_for_classification` reduces the transcript to user intent:
   - Tool-result blocks are dropped (noise).
   - Assistant turns are truncated to their first line (≤300 chars).
   - User turns are kept in full.
   - The result is capped at `LAV_CLASSIFY_MAX_CHARS` (default **12000** chars).
   This is why `classification` reflects what the *user* did, not assistant behavior.

3. **Classification call** — dispatched by backend (`LAV_CLASSIFY_BACKEND`: `auto`/`openai`/`ollama`):
   - **OpenAI path** (`openai_strict.py`): uses native `response_format: json_schema` with `strict: True`, so the model output is guaranteed to match `CLASSIFICATION_SCHEMA`.
   - **Ollama path** (`ollama_compat.py`): compatibility path for local models; output parsed leniently (`_parse_json_response` strips code fences and `<think>` tags).
   - Default model: `gpt-4.1-mini` (`LAV_CLASSIFY_MODEL`). Token budget 2000.

4. **Sanitization** — `_sanitize_result` enforces all schema invariants (see [schema.md](schema.md)): field presence, type coercion, vocabulary validation, `topics` cap of 5, and the `people → internal` escalation.

5. **Indexing** — the merged payload (provenance + sanitized semantic fields) is written to the Qdrant collection alongside the embedding vector.

## Language handling

`LAV_CLASSIFY_LANGUAGE` (default `en`) controls the natural-language fields only:

- When set to a non-English language, `summary`, `abstract`, and `process` are written in that language.
- All controlled / entity fields — `topics`, `people`, `clients`, `classification`, `data_sensitivity`, `sensitive_data_types` — **always stay in English**, keeping filters and vocabularies stable across languages.

This is why observed records mix Italian prose (`summary`/`abstract`) with English topic keywords and class labels.

## Field ordering as implicit reasoning

`CLASSIFICATION_SCHEMA` is ordered deliberately: descriptive fields (`summary`, `abstract`, `process`, `topics`, `people`, `clients`) precede the decision fields (`classification`, `data_sensitivity`, `sensitive_data_types`). Because the model emits JSON in order, it establishes context before committing to the controlled-vocabulary judgments — a built-in chain-of-thought without a separate reasoning pass.

## Failure & gap behavior

- **Empty interaction** → `summary` set to `(empty interaction)`, other fields default.
- **Model omits a field** → filled from `_EMPTY_RESULT` defaults.
- Observed gaps in live data (empty `summary`, empty `tools_used`) reflect either older indexing runs or interactions that produced no extractable user text. The code defaults prevent missing keys, but cannot retroactively enrich a record — a `--full` reindex is required to refresh them.
