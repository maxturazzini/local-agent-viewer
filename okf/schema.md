# Interaction Metadata Schema

Every interaction indexed into the LAV knowledge base carries a single flat metadata payload (the Qdrant point payload). The payload has two origins:

- **Provenance fields** are filled deterministically by the indexer from the canonical SQLite DB.
- **Semantic fields** are produced by the LLM classifier (see [enrichment-pipeline.md](enrichment-pipeline.md)) and then sanitized.

# Schema

## Provenance fields

These are factual, deterministic, and never invented by a model.

| Field | Type | Source | Description |
|-------|------|--------|-------------|
| `session_id` | `string` | DB | Unique interaction identifier. UUID for native sessions; prefixed for imports (e.g. `chatgpt:678778e3-...`). |
| `project` | `string` | DB | Workspace name the session ran in (e.g. `miniMe`). |
| `user` | `string` | DB | Username who owned the session (e.g. `maxturazzini`). |
| `host` | `string` | DB | Hostname where the interaction happened (e.g. `macChia`, `minimacs`). |
| `source` | `string` | DB | Originating tool. One of: `claude_code`, `chatgpt`, `codex_cli`, `cowork_desktop`. |
| `timestamp` | `string` (ISO 8601) | DB | Timestamp of the first message in the interaction. |
| `indexed_at` | `string` (ISO 8601) | Indexer | When the interaction was written into Qdrant. |
| `message_count` | `integer` | DB | Number of messages in the interaction. Used by the `--min-messages` filter to skip trivial sessions. |
| `tools_used` | `array<string>` | DB | Tools invoked during the interaction. Frequently empty in practice — population is best-effort. |
| `tags` | `array<string>` | Manual | Manually applied tags. Searchable filter. Empty unless curated via `lav kb tags`. |

## Semantic fields

These are generated by the classifier model and constrained by sanitization (`_sanitize_result`). Field order in the schema is intentional: descriptive fields come first so the model builds context before committing to the controlled-vocabulary fields — an implicit chain-of-thought.

| Field | Type | Description | Constraints |
|-------|------|-------------|-------------|
| `summary` | `string` | One sentence: what the user did or asked. | Required. Language follows `LAV_CLASSIFY_LANGUAGE`. |
| `abstract` | `string` | 2–3 sentences: context, problem addressed, decisions made. | Required. Language follows `LAV_CLASSIFY_LANGUAGE`. |
| `process` | `string` | The concrete workflow the user is executing, named as an activity (e.g. `debug deploy pipeline`, `client workshop preparation`). | Required field; **empty string** when not inferable. Language follows `LAV_CLASSIFY_LANGUAGE`. |
| `topics` | `array<string>` | Up to 5 specific keywords. Generic terms discouraged. | Hard-capped at 5 by sanitization. |
| `people` | `array<string>` | Third-party names mentioned. **Excludes** the user and the assistant. | Empty array if none. Non-empty forces `data_sensitivity ≥ internal`. |
| `clients` | `array<string>` | Companies or clients mentioned (e.g. `Coop Atto 2`, `Salvagnini`). | Empty array if none. |
| `classification` | `string` (enum) | Primary classification of the interaction. | One of the values in [classification.md](classification.md). Invalid → defaults to `development`. |
| `data_sensitivity` | `string` (enum) | Sensitivity level of the data discussed. | One of the values in [sensitivity.md](sensitivity.md). Invalid → defaults to `internal`. |
| `sensitive_data_types` | `array<string>` | Categories of sensitive data present. | Empty when `data_sensitivity = public`. See [sensitivity.md](sensitivity.md) for the vocabulary. |

## Sanitization guarantees

The `_sanitize_result` function enforces these invariants on every record before it is stored, regardless of model output:

- All nine semantic fields are always present (missing → schema default).
- Scalars coerced to string, lists coerced to array.
- `classification` not in the controlled vocabulary → `development`.
- `data_sensitivity` not in the controlled vocabulary → `internal`.
- `topics` truncated to the first 5.
- If `people` is non-empty and `data_sensitivity` is `public`, it is escalated to `internal`.

## Searchable filters

Via the MCP `semantic_search` tool and the `lav kb search` CLI, these payload fields are usable as filters:

- `classification`
- `tags`
- `project`

# Examples

A development session on the server, with client context:

```json
{
  "session_id": "09099fa8-3182-408f-9ae0-e77868b263d0",
  "project": "miniMe",
  "user": "maxturazzini",
  "host": "macChia",
  "source": "claude_code",
  "timestamp": "2026-04-22T07:51:11.818Z",
  "indexed_at": "2026-04-22T12:00:08.349722",
  "summary": "Prepared a structured recap of past Week 4 AI Team sessions.",
  "abstract": "The user gathered errors, homework and recurring patterns from prior Week 4 sessions to prepare today's session with Coop Atto 2. The assistant ran semantic searches across LAV, Obsidian and local files and compiled a concise report.",
  "process": "client workshop preparation",
  "topics": ["semantic search", "AI Team sessions", "Week 4 session", "homework assignments", "error analysis"],
  "people": ["Elena Segattini"],
  "clients": ["Salvagnini", "Confindustria Verona", "Coop Atto 1", "Coop Atto 2"],
  "classification": "analysis",
  "data_sensitivity": "internal",
  "sensitive_data_types": ["client_strategy", "internal_architecture"],
  "tags": [],
  "message_count": 70,
  "tools_used": []
}
```
