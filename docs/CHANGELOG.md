LAV-74: Codex parser — correct incremental watermark + source attribution from `originator`.
- **Watermark bug fixed** (`lav/parsers/jsonl.py:parse_codex_sessions`): the cursor was read per-event and compared as a **string**, then persisted as `datetime.now()` (naive local) instead of the max observed event. Codex events are UTC (`…Z`), so the lexical compare between a naive-local watermark and UTC events silently **dropped recent sessions**. Now: watermark loaded **once per project/host**, compared as timezone-aware **UTC** `datetime`, legacy naive watermarks interpreted as local time, and `parse_state` written **once at end of pass** as the true max imported timestamp. Helpers `_parse_codex_event_ts` / `_parse_codex_watermark_ts`.
- **`full_reparse` now honored**: previously ignored by the Codex parser. `--full` ignores the watermark AND wipes the reparsed sessions' Codex rows first (scoped by `codex:`-prefixed `session_id` + host) so re-import can't duplicate rows in tables without a UNIQUE constraint (`mcp_tool_calls`; `bash_commands`/`file_operations` already guarded).
- **Source attribution** (`config.py`, `jsonl.py`): `source` derived from `session_meta.payload.originator` (the real surface) instead of forcing `codex_cli`. Mapping: `codex_work_desktop`→`chatgpt_work_desktop`, `Codex Desktop`→`codex_desktop`, `codex_vscode`→`codex_vscode`, CLI variants/absent→`codex_cli`, unknown→`codex_local` (raw originator kept in `meta_json`). New `map_codex_source()`; new constants `SOURCE_CHATGPT_WORK_DESKTOP`/`SOURCE_CODEX_DESKTOP`/`SOURCE_CODEX_VSCODE`/`SOURCE_CODEX_LOCAL` + `CODEX_SOURCE_ALIASES`.
- **In-place re-attribution**: `upsert_session_source` gains `override_sources` — the Codex parser upgrades a stale `codex_cli` row to a newly recognized surface without affecting other parsers (they pass `None`). Runs on every parse (incremental included), since `session_meta` isn't gated by the watermark.
- **Sync routing** (`server.py`): a `scope=source` sync on any Codex surface (`codex_cli`/`chatgpt_work_desktop`/`codex_desktop`/`codex_vscode`/`codex_local`) triggers the same local parser; `codex_cli` stays the alias for compatibility.
- **UI**: friendly labels for the new sources in `dashboard.html`/`interactions.html` (filters were already value-dynamic — no schema/API change).
- **Test**: `tests/test_codex_source_parsing.py` (21 checks) — UTC-vs-legacy-local watermark, multi-rollout max watermark, incremental no-dup, originator mapping + unknown fallback, stale-`codex_cli` override (and non-Codex left untouched), `--full` ignores watermark + no dup.
- **Validated** on a copy of the prod DB: the 122 `codex_cli` sessions split into 53 `codex_desktop` + 34 `codex_cli` + 31 `codex_vscode` + 4 `chatgpt_work_desktop`; other sources unchanged.
- **Backfill (last 72h)** — separate deploy step to recover sessions dropped by the old watermark: back up the SQLite DB; find rollouts with ≥1 event in the last 72h; extract their session ids; delete only those sessions' raw rows (avoids dup in the unconstrained tool tables); `parse_codex_sessions(full_reparse=True)` over them so threads started before the window stay complete; then reclassify (`lav-classify`) + reindex Qdrant for the touched sessions. Verify per-source counts, Work threads visible, no dup, clean incremental rerun; deploy to macChia + minimacs per the runbook and restart the LAV python processes.

LAV-72: production hardening for the foundry classify runner.
- **Foundry client timeout/retry** (`foundry/client.py`): per-request deadline `LAV_FOUNDRY_TIMEOUT` (default 40s) + `LAV_FOUNDRY_MAX_RETRIES` (default 3, SDK-level on timeout/429/5xx). A bulk run had hung 13 min on a single Azure call with no client-side deadline — now impossible.
- **`lav-classify` foundry-aware** (`sql_classifier.py`): with `LAV_CLASSIFY_BACKEND=foundry` the runner now builds the client via `foundry.client.make_client()` (Azure endpoint + key) instead of the OpenAI default — previously a foundry-configured run would have called api.openai.com with the deployment name and 404'd. `OPENAI_API_KEY` is no longer required for the foundry backend.
- **Surgical reclassification**: new `lav-classify --meta-since <ISO-TS> --meta-model <LIKE>` selects ALREADY-classified rows by metadata `updated_at` / `model_used` (e.g. replace last-24h gpt-4.1 labels with deepseek) — the opposite of the incremental default.

LAV-72: Azure AI Foundry classifier backend (deepseek) + tightened taxonomy + input cleaning + sensitivity floor.
- **New backend `foundry`** (`lav/classifiers/foundry/`): `client.py` builds an OpenAI-SDK client for an Azure AI Foundry endpoint (`LAV_FOUNDRY_ENDPOINT`/`_KEY`/`_API_VERSION`, per-deployment overrides); `classify.py` does the full single-call strict-json_schema classification (reuses `openai_strict.SYSTEM_PROMPT` + `CLASSIFICATION_SCHEMA` so the task never drifts), with Foundry quirks handled: `max_completion_tokens` headroom for reasoning models, optional `LAV_FOUNDRY_REASONING_EFFORT`, json_schema→json_object fallback, and per-call `usage` capture. Dispatcher gains `LAV_CLASSIFY_BACKEND=foundry` (opt-in, never auto-selected).
- **Decision**: benchmarked 4 Foundry models on golden_set_v2 (98) — **deepseek-v4-flash wins** (classification ~68%, +9pt over gpt-4.1-mini; €0.00063/task ≈ €10.6 for the full 17k backfill; reasoning 0). gpt-5.1 is the EU-DataZone alternative (~62%, ~10× the cost). gpt-oss-120b excluded (43% classification). Residency: staying on deepseek (Global) is coherent with the already-US stack; zero-retention available via Modified Abuse Monitoring.
- **Taxonomy tightened** (`taxonomy.json` private + `taxonomy.example.json` public mirror, categories unchanged so the gold stays valid): positive discriminators replace the self-sabotaging "analysis is broad — prefer others"; fixed LinkedIn appearing in both marketing+content_production (now split by INTENT: inform→content_production, sell→marketing); analysis anchored to "examine an EXISTING artifact"; support anchored on error/broken; marketing-vs-operations = win-business-vs-internal-admin.
- **Input cleaning v2** (`prepare_messages_for_classification`): parses JSON-stringified content-block lists (the model saw raw JSON before) and strips recurring IDE/harness prose (`_NOISE_RE`); a 315k-char session now sends dense signal in its 12k head instead of wrapper noise. New `full_scan_text()` = uncapped concat for the detectors.
- **Sensitivity floor** (`sensitivity_floor`/`apply_sensitivity_floor`, opt-in `LAV_SENSITIVITY_FLOOR=1`): deterministic MINIMUM from regex detectors (credentials/keys/IBAN, multilingual health incl. `genetico`, `sales/`|`clients/` paths) + extracted entities + classification (content_production→internal, meeting/marketing→confidential), combined as MAX(floor, model) so it only RAISES sensitivity — the field's costly error is UNDER-escalation. On the 13 residual leaks the floor eliminates every sensitive→public miss.
- **Eval** (`tests/evals/eval_classify.py`): `foundry:<deployment>` routing; directional **leak-rate** metric for data_sensitivity (under/over/correct + severe) — accuracy is the wrong metric for a cautious system; per-model token+€/task cost table from `usage` (Azure Monitor lags on the /openai/v1 route); `PRICE` dict with deepseek + gpt-5.1 rates.
- **Not shipped to prod yet**: bulk backfill of the 17k is a separate deploy step (throughput bound ~14h at deepseek's 20-RPM GlobalStandard quota → request a bump or run overnight/resumable); auto-classify stays disabled until then.
- **Verified**: floor unit tests + offline re-application to the 13 real leaks (3 fixed, 2 improved, sensitive→public eliminated); eval token/cost table + leak metric; all Foundry deployments smoke-tested. Removed the stray `lav/taxonomy_v1.json` A/B baseline (gitignored).

LAV-68: Stabilize `detect_host()` — stop creating duplicate/corrupted host records.
- **Problem**: host identity was derived from `socket.gethostname()`, which on macOS is volatile — depending on network/Bonjour/DHCP state it returns the canonical name, the generic `Mac`, or an undecodable byte string that surfaces as mojibake (`U+FFFD`) / surrogate-escaped chars. `_normalize_hostname()` only stripped `.local`/`.localdomain`, so every transient value became its own `hosts` row and split one machine's sessions across host ids (~1600 cross-host duplicate messages). Observed on **macChia** (`Mac`, corrupted) and **minimacs** (`Mac`, corrupted) — each node's own parser hits it.
- **Fix** (`lav/parsers/jsonl.py`, `lav/server.py`):
  - New `_is_valid_hostname()` rejects empty, generic-fallback (`mac`/`imac`/`localhost`/…), control chars, surrogates, and `U+FFFD`.
  - New `_canonical_hostname()` with precedence `LAV_HOSTNAME` env → `config.json` `hostname` key → validated `socket.gethostname()` → `unknown` sentinel; a corrupted/generic detected name is discarded, never turned into a host record. `detect_host()`/`detect_host_from_path()` use it.
  - `ingest_remote_sessions()` validates the remote hostname too (a pre-fix agent can't pollute the collector), and `/api/export` ships the canonical hostname so cross-node identity is stable.
- **Config**: set a stable `"hostname"` key in each node's `config.json` (macChia → `macChia`, minimacs → `miniMacs`); documented in `config.*.example.json`.
- **Verified** (macChia): unit checks for the validators; monkeypatched volatile `socket.gethostname()` (`Mac`, mojibake, surrogates, `localhost`) all resolve to `macChia` via config; real incremental `lav-parse` attaches every project to `macChia (id=2)`, hosts stay `{1,2,8}`, no new host rows, duplicate session_ids stay 0, `PRAGMA quick_check` ok. macChia DB was first cleaned by hand (relabel host {3,4,5,6,7,9}→2 + keep-fuller dedup of 152 dup session_ids; report at `~/.local/share/local-agent-viewer/hosthygiene-report-20260705.txt`).
- **Follow-up (separate)**: one-time cleanup of the minimacs DB (keep `miniMacs`=4 and `cloud`=5, consolidate `Mac`+corrupted, dedup 128 sessions keep-fuller); consider a `lav hosts merge <src> <dst>` maintenance command.

LAV-66: Workflow/Task subagents get their own child conversation; parent identity no longer corrupted; display cleaned of system tags.
- **Problem (bug #3)**: current Claude Code writes every `subagents/**/agent-*.jsonl` record (both Task- and Workflow-spawned) with the **parent's** `sessionId`; only `agentId` distinguishes the subagent. The parser keyed everything by `sessionId`, so all subagent messages collapsed into the parent interaction: no child rows in `interactions`, parent `message_count`/`total_tokens` inflated with the whole tree (reference session `cb76ec06…`, Arcipelago: 1905 msgs / 95M tokens instead of 861 / 85M + 35 children). Measured blast radius on macChia: **748 parent sessions across 26 projects, 7184 agent files, 0 old-style files** with their own sessionId.
- **Problem (bug #4, consequence)**: `session_agent_info` was keyed by that shared sessionId, so the **parent's** `interactions` row got a subagent's `agent_id` and then a bogus `parent_session_id` from the ±120s proximity fallback in `resolve_agent_parents`.
- **Problem (bug #5, minor)**: `interactions.display`/`summary` could be an injected system wrapper (`<local-command-caveat>`, `<command-name>`, `<ide_opened_file>`, …) instead of the first real user text.
- **Fix** (`lav/parsers/jsonl.py`, `lav/queries.py`):
  - `parse_project` rewrites the identity of every agent-file record to a **synthetic session id `<parent>::agent-<agentId>`** (parent = the record's own `sessionId`) before any processing, so the whole pipeline (messages, token_usage, tool tables, session_sources, `update_interaction`) keys the subagent as a separate conversation with `parent_session_id`/`agent_id` set — same hierarchy the UI already renders (rollup, `↳ N agents`). Applies to **all three layouts**: `subagents/agent-*`, `subagents/workflows/wf_*/agent-*` and the older **root-level `<proj>/agent-*.jsonl`** (3.7k files on macChia — all verified to reuse a sibling parent's sessionId; none carries its own). Parent rows stay clean (bug #4 fixed for free).
  - **Heal query** at end of each project parse: a non-agent session whose `agent_id` belongs to one of its own children is the bug-#4 collapse signature. "Husk" parents (their own `.jsonl` is empty/pruned, the row existed only because subagent messages were collapsed into it) have no messages, so `update_interaction` early-returns and would never clean them — the heal clears `agent_id`/`parent_session_id` and re-derives their counts.
  - **`NOT EXISTS` idempotency guards** on the three child tables without a UNIQUE constraint (`bash_commands`, `search_operations`, `mcp_tool_calls`), both in `process_tool_call` and in `ingest_remote_sessions`: agent files are reprocessed on every incremental run (LAV-65), which duplicated these rows on every 15-min parse since LAV-65 (pre-existing; amplified by the re-keying).
  - **Grouped search fix** (`lav/queries.py get_interactions_list`): in grouped mode (web UI default) child sessions are hidden, so an FTS match living only in a subagent transcript — or a pasted child session id — returned nothing. Search now also surfaces masters that have a matching child (CLI/MCP search stays flat, unchanged).
  - **Only hex-id agent files are subagents** (followup): Claude Code writes several kinds of `agent-*.jsonl` that all reuse the parent's sessionId. Real Task/workflow subagents carry a single hexadecimal id (`a1e94cf`, `abb5998eb0fb92919`); **meta artifacts** carry a non-hex first segment — `acompact-…` (auto-compaction checkpoints, 169 files), `aprompt_suggestion-…` (prompt suggestions, 267), `aside_question-…` (side questions, 5). The initial rekey turned all of them into phantom child conversations: `acompact`/`aside_question` duplicate the parent's own messages (double-counted tokens — 5299 dup messages / 169 phantom children measured on macChia), `aprompt_suggestion` adds internal noise. New `is_real_agent_id()` gates on a hex-only first segment; the parser skips every meta file, `purge_meta_children()` sweeps pre-fix phantoms each parse, and `ingest_remote_sessions` drops any non-hex `::agent-` bundle defensively. Verified on AIMaxViewer (phantom children 10→0, real subagents 50 untouched, incremental clean) and globally on macChia: duplicated messages on the real host dropped from ~123 to 4 (the residual 4 are one parent whose own `.jsonl` was pruned; the rest of the historical dup count sits under legacy phantom host rows — a pre-existing, separate DB-hygiene issue, not LAV-66).
  - New **`messages.agent_id` column** (+ `_migrate_add_agent_id`, partial index `idx_msg_agent`), populated from `message.agentId` on parse and propagated through `/api/export` → `ingest_remote_sessions`.
  - `--full` wipe extended from messages/token_usage to **all child tables** (file_operations, bash_commands, search_operations, skill_invocations, subagent_invocations, mcp_tool_calls): removes rows misattributed to the parent by the pre-fix parser and stops the unbounded duplication of the UNIQUE-less tables (bash_commands, search_operations, mcp_tool_calls) on every `--full`.
  - `--full` wipe is now **restricted to sessions whose source files still exist on disk** (session ids derived from filenames, incl. the synthetic agent ids). Previously it deleted messages/token_usage for *every* claude_code session of the project — Claude Code prunes files after `cleanupPeriodDays`, so sessions with pruned files were silently emptied and never rebuilt (latent data-loss since LAV-39; on minimacs the DB holds sessions back to 2026-02 while the oldest local file is 2026-06).

## 0.1.5 — 2026-04-23

LAV-39: Fix double-counting of Claude Code costs/tokens on multi-block turns.
- Claude Code writes one JSONL record per content block (`thinking`, `tool_use`, text) but all blocks of a single API turn share the same `message.id` and `usage`. The parser was summing usage per row, inflating `cost_usd` and `total_tokens` by ~20–35% on Opus sessions with thinking.
- `messages` and `token_usage` gain an `api_message_id` column; new partial `UNIQUE(session_id, project_id, api_message_id)` index on `token_usage` makes `INSERT OR IGNORE` idempotent per API turn.
- `process_message_content` credits `tokens_in/tokens_out` only to the first block of each `api_message_id` (checks DB for existing non-zero row) so `update_interaction`'s `SUM(tokens_in+tokens_out)` stops double-counting.
- `lav-parse --full` now wipes prior claude_code `messages`/`token_usage` rows for the project+host before re-parsing, so old duplicates with empty `api_message_id` are cleaned up on the authoritative reparse.
- `get_session_cost_profile` exposes `api_message_id` in each timeline entry.
- After deploying the fix, run `lav-parse --full` once to rebuild existing data cleanly.
